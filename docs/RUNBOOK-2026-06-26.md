## Local â†” GitHub sync (binding)

**Goal:** your checkout always matches `origin/main` before you run the app.

| Situation                 | Command                                                 | What it does                                                     |
| ------------------------- | ------------------------------------------------------- | ---------------------------------------------------------------- |
| Start dev (automatic)     | `bun dev`                                               | runs `bun run guard` first via `predev`                          |
| Check sync only           | `bun run guard`                                         | ff-down if behind; push if ahead; never destroys work            |
| Pull only (ff-down)       | `bun run pull`                                          | fast-forward to `origin/main` when clean                         |
| Behind + want GitHub wins | `git fetch origin main && git reset --hard origin/main` | **destructive to local-only commits** â€” use when diverged        |
| Stuck rebase / diverged   | `bun run guard --force`                                 | preserves tip to `recovery/guard-*` then resets to `origin/main` |
| Opt out of auto-sync      | `CQM_NO_SYNC=1 bun dev`                                 | skip guard on dev start                                          |

**Symptoms:**

- Renamed docs missing locally â†’ you are **behind** `origin/main`. Run `bun run pull` or `bun run guard`.
- `git status` shows dozens of `M` files with empty diff â†’ CRLF phantom (Windows). `git reset --hard origin/main` re-checkouts as LF.
- `sync-guard: stuck rebase` on a clean tree â†’ stale rebase state; `bun run guard --force` or `git rebase --abort` if a real rebase is active.

Post-commit hook auto-pushes **HEAD â†’ `origin/main`** (opt out: `git config hooks.autopush false`). Work on `main` only â€” no PRs.

## Deploy (GitHub Pages, via Actions)

Pushing to `main` triggers `.github/workflows/pages.yml`, which builds and deploys to GitHub Pages.
Live site: <https://0thernes.github.io/cosmogonic-quantum-mechalogodrom/>.

**GitHub repo hygiene (one-person, direct-main, no PRs):**

- **Truth branch:** `main` only. The repository hook runs the full gate before commit and then pushes
  `HEAD:main`; do not open PRs or leave automation branches as alternate truth.
- **CI:** push to `main` runs `.github/workflows/ci.yml` (Linux + Windows full gate) + `pages.yml` (deploy).
  Diagnose: `gh run list --branch main --limit 5` Â· `gh run view <id> --log-failed`.
- **Pages gotcha:** `github-pages` environment must allow `main` deploys (see below).

- **Gotcha (already fixed, keep in mind):** the `github-pages` _environment_ deployment-branch policy
  must allow `main`, or every deploy is silently **rejected** at the deploy step while CI stays green.
  Re-apply with:
  `gh api --method POST repos/0thernes/cosmogonic-quantum-mechalogodrom/environments/github-pages/deployment-branch-policies -f name=main -f type=branch`
- Diagnose a stale deploy: `gh run list --workflow=pages.yml` (look for `failure`), then
  `gh run view <id>` and read the **annotations** (the env-protection rejection shows there, not in the logs).

## Roll back

- **Code/release:** `git checkout vX.Y.Z` (automated releases use lightweight tags) â€” the last known-good is the
  newest `v*` tag. To re-deploy an older state, push that state to `main` (Pages always deploys `main`).
- **Pages only:** the GitHub Pages "deployments" list keeps prior successful deploys; the previous one is
  the rollback target if a bad deploy ships.
