# A. Pre-2016 Game & Simulation AI — A Cited Dossier

Every technique below predates the transformer/attention era (2017) and the 2016 generative wave. For each: one-line definition, era + landmark title, compute cost at ~100–1000 agents, and concrete wiring into THIS sim at **micro** (per-agent) and **macro** (tribe/economy) scale. Determinism traps are flagged (float-order, `Map`/`Set` iteration, `Date.now`).

## A.1 Finite-State Machines + Hierarchical FSMs

- **Definition:** An agent is in exactly one of N discrete states; edges fire on conditions/events. HFSMs nest sub-machines inside states so shared logic (flee, regroup) is reused across parents.
- **Era / landmark:** The workhorse of 1990s–2000s game AI (Pac-Man ghosts, Quake monsters, _Half-Life_ soldiers). Damian Isla frames Halo 2's brain as "a hierarchical finite state machine (HFSM) or a behavior tree." ([Isla, GDC 2005](https://www.gamedeveloper.com/programming/gdc-2005-proceeding-handling-complexity-in-the-i-halo-2-i-ai))
- **Cost @ 100–1000:** Effectively free — one integer state + a switch per agent per tick. O(agents). 1000 agents is noise.
- **MICRO:** Give each of the up-to-10k organisms a tiny `state: u8` packed into a typed array (Idle/Forage/Flee/Mate/Rest/Hunt — the existing 26 behaviors become the leaf actions; the FSM is the _arbiter_ that picks which is active). Transitions read needs + perception; resolve ties with the seeded stream, never `Math.random`.
- **MACRO:** The HFSM parent layer = the **tribe**: a Louvain community is itself a state machine (Peaceful → Alarmed → Migrating → War) whose state biases every member's transition table.
- **Determinism:** iterate states by fixed index order, never object insertion order.

## A.2 Behavior Trees (Halo 2 era)

- **Definition:** A tree of composable nodes — Selector (first child that succeeds), Sequence (all in order), Decorator, Leaf-action — re-ticked each frame; replaces FSM edge-spaghetti with structured priority.
- **Era / landmark:** **Halo 2 (2004)**, Damian Isla, GDC 2005; ~50 behaviors as a **behavior DAG** selected by **prioritized lists** with binary relevancy tests, plus **impulses** and **stimulus behaviors**. Pooling trick: state cost "only grows with the maximum depth of the tree, not the number of behaviors" (~25K for 100 actors). ([Isla 2005](https://www.gamedeveloper.com/programming/gdc-2005-proceeding-handling-complexity-in-the-i-halo-2-i-ai); [BT Starter Kit, GameAIPro ch.6](https://www.gameaipro.com/GameAIPro/GameAIPro_Chapter06_The_Behavior_Tree_Starter_Kit.pdf))
- **Cost @ 100–1000:** Cheap but heavier than FSM — depth×branching node-visits/tick. Mitigate with bitvector relevancy tags, event-driven stimulus behaviors instead of polling, and **time-slicing** (tick 1/k of the population each frame). 1000 agents × ~10-node trees ≈ 10k node-evals/frame, trivial.
- **MICRO:** A single **shared, immutable tree definition** (data, not per-agent objects) with per-agent blackboard pooled by depth (Isla's model — fits the instanced/typed-array design). **Shoggoth predators** get `Selector(Engage ▸ Stalk ▸ Patrol)`.
- **MACRO:** The **puppet-master event NPCs** are the canonical BT users: rich, authored, directable via Isla-style "orders"/"styles" (allow/disallow lists).
- **Determinism:** visit children in array order; any "random child" pulls from the seeded stream.

## A.3 GOAP — Goal-Oriented Action Planning

- **Definition:** Agent holds a symbolic world-state + goal; an A\* planner chains **actions** (precondition→effect, each with a cost) backward from goal to current state, producing a plan it executes.
- **Era / landmark:** **F.E.A.R. (2005)**, Jeff Orkin (Monolith), adapting STRIPS (1971). ~120 actions, ~70 goals, plans usually 1–2 actions, A\* over world-states with action costs as heuristic. ([Orkin, "Three States and a Plan"](https://gdcvault.com/play/1013282/Three-States-and-a-Plan); [Game Developer writeup](https://www.gamedeveloper.com/design/building-the-ai-of-f-e-a-r-with-goal-oriented-action-planning))
- **Cost @ 100–1000:** The pricey one. A\* per replan; cost ∝ branching^plan-length. For 1000 agents you **must** keep action sets small (≤20), cache plans, replan only on world-state change, and budget N replans/frame.
- **MICRO (select only):** GOAP-lite for "elite" organisms: 3–4 actions, plan length 1–2.
- **MACRO:** Ideal for the **10 game-theoretic titan minds** + the **puppet-master**. Goal (`tribe_dominant`, `predator_culled`, `resource_secured`) + actions (`Ally`, `Betray`, `SpawnShoggoth`, `SeedFood`) with preconditions over macro state (PageRank, tribe sizes, weather). The titan's iterated-PD/replicator choice becomes the planner's _cost function_.
- **Determinism:** A\* tie-breaking must be deterministic — break ties by a stable key (action index), never by hash/insertion order; never key the open-set on a JS `Map` whose iteration order you rely on.

## A.4 Utility / Needs-Based AI (The Sims) — RECOMMENDED MICRO DEFAULT

- **Definition:** Each candidate action gets a scalar **score** = how much it satisfies current needs; pick highest (or softmax-sample). In The Sims the _objects advertise_ their utility ("smart objects"), so intelligence lives in the environment.
- **Era / landmark:** **The Sims (2000)**, Will Wright. Six motives; an object "broadcasts all the ways it can change a Sim's utility level," combined with the Sim's deficit to score actions. ([Utility system](https://en.wikipedia.org/wiki/Utility_system); ["Smart Objects," Gaslamp](https://archive-gaslamp.dredmor.com/2015/04/15/smart-objects-or-everything-i-know-about-ai-i-stole-from-the-sims/))
- **Cost @ 100–1000:** Very cheap and the most _scalable_ believable-AI technique: per agent, score K candidate actions = K multiply-adds + a max. K≈10 × 1000 agents = 10k evals/frame. Response curves are LUTs. No search.
- **MICRO (primary driver):** Maintain a small **needs vector** per organism (energy, safety, reproduction, social/tribe-affinity, curiosity) in typed arrays. Make **food, water, mates, shoggoths, tribe-center, RD-rich tiles** into "smart objects" that advertise utility deltas within a radius. Each tick the organism scores its 26 behaviors via response curves and picks argmax. Dovetails with FSM (utility picks the state) and steering (utility picks the target).
- **MACRO:** Smart-object advertising scales to macro — a tribe-center or titan stronghold advertises affinity/threat over a wide radius via the influence-map textures (§A.6).
- **Determinism:** softmax/weighted sampling **must** draw from the seeded stream; argmax ties broken by behavior index. **Single highest-ROI technique for the micro layer.**

## A.5 Steering Behaviors + Boids (Reynolds 1987)

- **Definition:** Movement = vector sum of simple urges (seek, flee, arrive, wander, separation, alignment, cohesion, obstacle-avoid) as steering force on a point-mass. Flocking = the 3 boid rules over **local** neighbors only.
- **Era / landmark:** **Boids, Craig Reynolds, SIGGRAPH '87** ("Flocks, Herds, and Schools"); steering taxonomy at GDC 1999. Three rules: **separation** "steer to avoid crowding local flockmates," **alignment** "steer towards the average heading," **cohesion** "steer toward the average position." ([red3d.com/cwr/boids](https://www.red3d.com/cwr/boids/); [SIGGRAPH87 PDF](https://www.red3d.com/cwr/papers/1987/SIGGRAPH87.pdf))
- **Cost @ 100–1000:** Naïve neighbor search is **O(n²)**. With a **spatial grid/hash** it drops to "nearly O(n)" — the critical optimization. The sim already maintains a spatial hash (`src/math/spatial-hash.ts`, rebuilt every 2nd frame).
- **MICRO:** Locomotion substrate beneath utility/BT: utility/BT decides _what_ (forage vs flee), steering decides _how to move_. Seek utility-chosen targets, flee shoggoths (predator avoidance = `flee` weighted by fear-need).
- **MACRO:** Wire separation+cohesion+alignment **per tribe** (Louvain membership = flock id) for legible herd/migration dynamics.
- **Determinism:** accumulate neighbor contributions in a **fixed iteration order** (sort neighbor indices, or iterate grid cells in index order) so float summation is reproducible across runs/machines. **#1 determinism pitfall in flocking.**

## A.6 Blackboard Architectures; Influence Maps

- **Definition (blackboard):** Shared data structure that independent "knowledge sources" read/write to cooperatively solve a problem; a controller schedules writes. **Influence map:** a 2D grid where each agent stamps a falloff "influence" (threat, ownership, scent); summed cells give cheap spatial reasoning (frontlines, safe routes, contested zones).
- **Era / landmark:** Blackboard from **Hearsay-II** (1970s, CMU); influence maps an RTS staple since the 1990s. ([Nii, "Blackboard Systems," Stanford CS-TR-86-1123](http://i.stanford.edu/pub/cstr/reports/cs/tr/86/1123/CS-TR-86-1123.pdf))
- **Cost @ 100–1000:** Blackboard: cheap (shared memory + scheduler). Influence maps: O(cells) to decay/blur + O(agents) to stamp; a 128×128 map blurred each tick is GPU-trivial and **maps onto a WebGL texture** (ping-pong FBO, same as the existing reaction-diffusion in `src/sim/reaction-diffusion.ts`).
- **MICRO (perception):** Organisms sample threat/food/territory textures in their step for near-free spatial awareness — flee high-threat, seek high-food.
- **MACRO:** Run **influence maps as textures** alongside the RD ground: a "threat" map (shoggoths + titan aggression, blurred + decayed), a per-tribe "territory" map, a "scent/food" map. The **blackboard** is the tribe/titan coordination bus: titans post intents, tribes post alarms, puppet-master posts events. This delivers the requested **layered, reactive** behavior — micro agents react to macro fields without per-pair communication.
- **Determinism:** blackboard writes within a tick must be order-independent or applied in fixed priority order (define a knowledge-source index).

## A.7 Fuzzy Logic; Decision Trees; Markov Chains

- **Definition:** **Fuzzy** — graded membership (0..1 "hungry"), fuzzy AND/OR, defuzzify; smooth transitions vs FSM's hard switches. **Decision trees** — authored/learned if-then splits → action. **Markov chains** — next state sampled from a transition matrix depending only on current state.
- **Era / landmark:** Fuzzy controllers widespread in 1990s–2000s; decision trees + fuzzy both in **Black & White (2001)**; Markov chains for ambient/procedural behavior. ([Pirovano, "Fuzzy Logic for AI in Games"](https://www.academia.edu/6850195/The_use_of_Fuzzy_Logic_for_Artificial_Intelligence_in_Games))
- **Cost @ 100–1000:** All three cheap, O(agents), sub-microsecond each.
- **MICRO:** **Fuzzy** is the smoothing layer over utility/FSM — a Fuzzy State Machine for mood (calm↔agitated↔panicked) driving steering gains. **Decision trees** = compact, _evolvable_ "species brains" mapping (need, threat, weather, tribe-state) → behavior (mutate split thresholds via the seeded stream → lineage variation).
- **MACRO:** **Markov chains** drive **weather/economy regime transitions** (Calm→Storm→Bloom) — ties directly to `src/sim/weather` and the chaos/entropy axis.
- **Determinism:** every defuzzification is deterministic; every Markov transition **must** sample the seeded stream; normalize rows in fixed order.

## A.8 Spreading-Activation / Associative + Episodic Memory

- **Definition:** Memory as a graph of concept nodes with weighted links; activating a node spreads decaying activation to neighbors (priming, recall). Episodic memory stores _specific events_ (what/where/when) distinct from semantic knowledge.
- **Era / landmark:** **Collins & Loftus (1975)** spreading activation; **Anderson's ACT-R** (base-level activation = recency+frequency); Tulving's episodic/semantic split. ([Spreading activation](https://en.wikipedia.org/wiki/Spreading_activation))
- **Cost @ 100–1000:** Tiny per-agent net (8–32 nodes, 1–2 hops) is cheap (O(edges)). A large shared semantic graph runs once per tick, not per agent.
- **MICRO:** Small associative memory per organism: nodes for {danger-here, food-here, ally, that-shoggoth, this-tile}. Meeting a shoggoth at a location spreads activation to the location node → the agent "remembers" and avoids it (decay = forgetting). Learning without neural training.
- **MACRO:** **Episodic traces** ("tribe X betrayed us at tick T near (x,y)") feed the titan PD memory and tribe grudges — the iterated-PD in `src/math/games.ts` already needs a memory of past defections; model it as episodic nodes with base-level decay.
- **Determinism:** activation decay is deterministic (watch summation order); any stochastic recall draws from the seeded stream.

## A.9 Perceptrons + Small MLPs; Threshold/Logic-Gate Neurons

- **Definition:** **McCulloch–Pitts (1943)** binary threshold neuron (fires if weighted inputs ≥ threshold) = a logic gate. **Perceptron (Rosenblatt 1958)** adds learnable weights + error-correction (linear only). **MLP** = input→hidden→output with nonlinearity, trainable by backprop (1986).
- **Era / landmark:** M-P 1943; Rosenblatt 1958; MLP/backprop Rumelhart-Hinton-Williams 1986. ([MLP](https://en.wikipedia.org/wiki/Multilayer_perceptron))
- **Cost @ 100–1000:** A tiny fixed MLP (8→6→4 ≈ 70 weights) ≈ 70 MACs/agent forward — **negligible**. 1000 agents ≈ 70k MACs/frame, far cheaper than the existing connectome. No training cost if weights are fixed/evolved. **Forward-only is the cheap, deterministic path.**
- **MICRO:** Formalize the existing per-organism "connectome" as a **fixed tiny MLP whose weights are genome-encoded**. Inputs = needs + sampled influence-map values + nearest-neighbor relations; outputs = behavior logits (feed the utility/FSM arbiter) or steering forces. Use **threshold/logic neurons** for ultra-cheap reflexes ("if threat>θ AND energy<φ → flee") in bit-ops.
- **MACRO:** A small shared net can score tribe-level decisions, but GOAP/Markov are the better macro fits; keep MLPs micro.
- **Determinism:** pure deterministic float math; fix the activation function (tanh/relu) and summation order; quantize for cross-platform bit-identity if needed.

## A.10 Neuroevolution (NEAT 2002), GAs, Evolution Strategies — CENTRAL TO LINEAGE

- **Definition:** Evolve neural-net **weights and topology** (or any genome) via selection + crossover + mutation against a fitness function — no gradients. **NEAT**: starts minimal, _complexifies_, innovation numbers for safe crossover, **speciation** to protect new structures. **GA** (Holland): bitstring genomes. **ES** (Rechenberg/Schwefel): real-valued vectors with self-adapting mutation.
- **Era / landmark:** **GA — Holland 1975**; **NEAT — Stanley & Miikkulainen 2002**; **rtNEAT (2003)** runs evolution _continuously in real time_, shipped in **NERO**. ([NEAT](https://en.wikipedia.org/wiki/Neuroevolution_of_augmenting_topologies); [Stanley CEC02](https://nn.cs.utexas.edu/downloads/papers/stanley.cec02.pdf); [Holland 1975](https://mitpress.mit.edu/9780262581110/adaptation-in-natural-and-artificial-systems/))
- **Cost @ 100–1000:** Inference = the tiny-MLP forward pass (cheap). Evolution is amortized: crossover+mutate at reproduction events, not every frame. **rtNEAT** replaces one low-fitness agent per K ticks (≈1 spawn/K frames) — built for real-time games. Cap topology growth to bound inference.
- **MICRO:** Genome encodes the §A.9 MLP (+ optionally body/behavior params). **Reproduction = crossover + seeded mutation**; fitness = lifespan + energy + offspring + tribe contribution.
- **MACRO:** **rtNEAT-style continuous replacement** delivers "offspring/lineage/social economy" without generational stalls. NEAT speciation maps onto **Louvain tribes** (species ≈ tribe). GA/ES also evolve macro params (titan strategies, weather regimes).
- **Determinism (make-or-break):** all selection, crossover points, mutations from the single `mulberry32` stream; **stable-sort the population by a stable key before selection** (never `Set`/`Map` order); fixed innovation-number counter; reproducible fitness summation. **The technique most likely to silently desync if the global RNG is touched.**

## A.11 Cellular Automata + Conway's Life; Reaction-Diffusion as Life

- **Definition:** **CA** — a grid updated in lockstep by a local rule over neighbors. **Conway's Life** (B3/S23): born at 3 neighbors, survive at 2–3; Turing-complete. **Reaction-diffusion** — two morphogens react+diffuse (Gray-Scott); Turing patterns (spots/stripes/spirals).
- **Era / landmark:** **Conway's Life, 1970** (Gardner); **Turing 1952** "Chemical Basis of Morphogenesis"; **Gray-Scott**. ([LifeWiki](https://conwaylife.com/wiki/Conway's_Game_of_Life); [Gray-Scott](https://biologicalmodeling.org/prologue/gray-scott))
- **Cost @ 100–1000:** Grid-bound, **not** agent-bound — O(cells), the most GPU-native technique here: one fragment-shader pass/tick on a ping-pong FBO. The sim already runs RD ground (`src/sim/reaction-diffusion.ts`, stepped every odd frame).
- **MICRO:** Use RD/CA fields as **inputs to the §A.9 micro-MLPs**, so environment literally drives brains.
- **MACRO / ENVIRONMENT:** Close the loop so the RD field is _alive_: RD concentration = food/biomass that organisms eat (deplete a channel) and fertilize (deposit), so the pattern co-evolves with the population — a true ecosystem feedback (this is the substrate for "rendering alters dynamics"). Add a **CA "weather/plague/fire" layer** whose live cells are hazards organisms flee via influence-map sampling.
- **Determinism:** CA/RD deterministic by construction — but enforce **strict double-buffering** (read A, write B, swap); seed any stochastic CA variant from the stream.

## A.12 Classic Artificial Life (the canon)

- **Tierra (Tom Ray, 1990–91):** Self-replicating machine-code organisms in a virtual CPU; evolved **parasites and hyperparasites**. _For you:_ a model for **code-as-genome** and emergent ecological roles (parasitism) without authoring them. ([Tierra](<https://en.wikipedia.org/wiki/Tierra_(computer_simulation)>); [Ray PDF](https://faculty.cc.gatech.edu/~turk/bio_sim/articles/tierra_thomas_ray.pdf))
- **Polyworld (Larry Yaeger, 1992/94):** Agents with **Hebbian, genome-encoded neural nets** take **rendered vision** as input; evolution trends toward complexity. _For you:_ blueprint for the micro-brain + vision-from-influence-maps loop at exactly this agent scale. ([Yaeger, PMC](https://pmc.ncbi.nlm.nih.gov/articles/PMC2801533/))
- **Framsticks (Komosinski & Ulatowski, 1996–):** Evolves both **body (sticks) and brain** in 3D physics. _For you:_ model for co-evolving morphology + controller if body variation is added. ([Framsticks](https://en.wikipedia.org/wiki/Framsticks); [VW2000 PDF](https://www.framsticks.com/files/common/Komosinski_FramsticksSimul_VW2000.pdf))
- **Creatures / Norns (Steve Grand, 1996):** A **952-neuron, ~5000-connection net in 9 lobes** + a **separate simulated biochemistry/"hormone" system** + a **genome encoding both brain and biochemistry**; norns breed and mutate. Ran on a 1996 PC. _For you:_ **gold-standard micro architecture** — net + chemistry + genome. ([Zucconi](https://www.alanzucconi.com/2020/07/27/the-ai-of-creatures/); [Grand et al., _Artificial Life_ 5(1)](https://direct.mit.edu/artl/article/5/1/77/2314/The-Creatures-Global-Digital-Ecosystem))
- **Black & White creature (Lionhead / Richard Evans, 2001):** A **Belief-Desire-Intention** agent that _learns by reinforcement_ using **decision trees + perceptrons**; reward/punishment attach to **beliefs, not actions**. _For you:_ template for a player-trainable "hero" organism or the **NHI beings** (click-to-launch). ([Postmortem](https://www.gamedeveloper.com/design/postmortem-lionhead-studios-i-black-white-i-); [Richard Evans](<https://en.wikipedia.org/wiki/Richard_Evans_(AI_researcher)>))
- **Spore creatures (Maxis / Chris Hecker, 2008):** **Procedural animation ("Spasm")** rigs/animates _never-before-seen_ morphologies via IK + authored-relative-to-body motions. _For you:_ the answer to animating evolved/variable bodies deterministically. ([Hecker](https://www.chrishecker.com/How_To_Animate_a_Character_You've_Never_Seen_Before); [Spore Animation White Paper](https://www.gameanim.com/2008/08/10/spore-animation-white-paper/))

## A.13 Recommended LAYERED architecture (woven MICRO ↔ MACRO, deterministic)

- **Environment layer (GPU, deterministic):** RD biomass + CA hazards + influence-map textures (threat/territory/scent). Drives everything; closes the eat/fertilize feedback loop.
- **Perception layer:** Spatial hash grid (shared by boids + neighbor queries) + sampling of influence/RD textures.
- **Micro-brain layer (per organism):** Genome-encoded tiny MLP / threshold neurons (Creatures/Polyworld-style) → **Utility/needs arbiter** (The Sims, _primary driver_) → smoothed by **fuzzy** mood → selects active **FSM state / BT leaf** → executed as **steering/boids** movement. Small **associative/episodic memory** for learned avoidance.
- **Macro layer:** **Blackboard** bus; **Louvain tribes ≈ NEAT species** as HFSMs; **10 titans run GOAP** with iterated-PD/replicator cost functions + episodic grudge memory; **Markov** regime transitions for weather/economy.
- **Evolution layer:** **rtNEAT-style continuous** crossover+mutation at reproduction events → lineage, speciation, social economy.
- **Determinism spine:** one `mulberry32` stream threads every stochastic choice; fixed iteration orders; double-buffered fields; stable sort before any selection.

# C. Safe Read+Run-Only AI Terminal Design (Bun + TypeScript)

## C.0 Threat model & core principle

The model is **untrusted**. Prompt injection from repo contents (a malicious README, a crafted test name, poisoned git history) can make it _propose_ anything. Therefore: **the model only proposes tool calls; the server is the sole authority that validates and executes them.** Every constraint below is enforced server-side. There is **zero reliance on prompt instructions or model self-restraint** — those are defense-in-depth at best, never a control (Plan-Then-Execute / Action-Selector patterns, arXiv 2506.08837).

Two independent walls protect the repo:

1. **Tool-surface wall** — the model can only call 4 narrow tools; none has a write capability.
2. **Execution wall** — `run_command` is gated by an allowlist+denylist validator and executed with **no shell**, so shell metacharacters are inert data, not operators.

## C.1 Tool surface

Exactly four tools. All paths repo-relative; the server resolves and confines them. No tool can create, edit, delete, move, or write.

| Tool          | Signature                           | Constraints (server-enforced)                                                                                                                                                           |
| ------------- | ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `read_file`   | `read_file(path, maxBytes?)`        | Path confined to repo root (§C.3). Refuses dirs, escaping symlinks, files > cap (default 256 KiB). Read-only `Bun.file()`.                                                              |
| `list_dir`    | `list_dir(path)`                    | Path confined. Names + type + size only. Does not follow symlinks out of root. Caps entries (~1000).                                                                                    |
| `grep`        | `grep(pattern, glob?, maxMatches?)` | Runs as an **argv array** via allowlisted `rg`/`git grep`, never a shell. `pattern` is one argv element — regex metachars are literal data to ripgrep. Confined to root. Output capped. |
| `run_command` | `run_command(argv: string[])`       | **argv is an array, never a string.** Passes the validator (§C.2). Only allowlisted programs+subcommands, no shell, confined cwd, timeout + output cap (§C.5).                          |

Design notes:

- `run_command` takes a **string array, not a command line** — the single most important API choice. No shell ⇒ `;`, `&&`, `|`, `>`, `` ` ``, `$()` are never interpreted; they would be literal args to a program that doesn't understand them (Auth0 / eslint-plugin-security / Semgrep all converge on array args + no shell as the root fix).
- `read_file`/`list_dir`/`grep` are **dedicated tools**, not `run_command(["cat", ...])` — common read paths bypass process spawning and keep the allowlist small.
- No `write_file`, no `apply_patch`, no stdin-from-model. The capability does not exist in the surface.

## C.2 Read-only enforcement: allowlist + denylist for `run_command`

### C.2a Allowlist (default-deny)

Default-deny: anything not matching an allowlist entry exactly is rejected. Keyed by `(program, subcommand)` with per-entry flag allowlists. Bare program names only — never a path, never `.bat`/`.cmd`/`.ps1`.

- `git`: read verbs only — `log, status, diff, show, rev-parse, branch --list/-a/-v, blame, ls-files, ls-tree, cat-file, shortlog`.
- `bun`: `test` (any non-write flags), `run` restricted to a **script-name allowlist** (`check, typecheck, tsc, lint, bench`), `pm ls`.
- `tsc` (`--noEmit`, `-p`, `--project`, `--pretty`, `--strict`), `eslint`/`oxlint` (**no** `--fix`), `prettier --check` (**never** `--write`), `rg`, `node --version`.

`bun run` is restricted to a script-name allowlist so the model can't invoke an arbitrary `package.json` script that writes/installs. (For higher assurance, resolve the script body and re-validate, or run under the §C.6 sandbox.)

### C.2b Denylist / hard guards (checked even for allowlisted entries)

- **Write/destructive git verbs:** `add, commit, push, pull, fetch, merge, rebase, reset, checkout, switch, restore, clean, rm, mv, stash, apply, am, cherry-pick, revert, gc, prune, worktree, config (write), update-ref, filter-branch, fast-import, replace, notes add`.
- **Install / package mutation:** `bun add|remove|install|update|link|unlink|patch|create`; `npm/pnpm/yarn/npx`, `pip`, `cargo add/install`, `go install`.
- **File-mutating coreutils / shells:** `rm, del, erase, mv, move, cp, copy, ren, rename, mkdir, rmdir, touch, ln, mklink, dd, truncate, tee, sed -i, chmod, chown, attrib, icacls, takeown`, and **any shell** (`bash, sh, zsh, cmd, cmd.exe, powershell, pwsh, wsl, env`).
- **Shell-metacharacter scan on every argv element** — reject `; & | > >> < $( ) \` { } && ||`/ newline/CR (the`grep` `pattern` element is exempted since it's data to ripgrep, never a shell). Because no shell runs, these are already inert; the scan is a **second wall** to catch any future code path that reintroduces a shell.
- **Output redirection to files:** reject `>`, `>>`, `1>`, `2>`, `&>`, and `-o/--output/--output-file/--log-file`-style write flags.
- **Network-mutating / exfil:** reject `curl, wget, Invoke-WebRequest, nc, ssh, scp, rsync, git remote add/set-url`.
- **Env-var injection:** reject `NAME=value` in the _program_ position; pass no model-supplied env (§C.4).

### C.2c Windows-specific hard rules (CVE-2024-27980)

Per the Node.js April 2024 advisory, on **Windows** args can break out of the intended command when the target is a **`.bat`/`.cmd`** file — even with `shell` disabled — because the OS routes batch files through `cmd.exe`, which re-parses the argument string. Mitigations:

1. **Never spawn a batch file.** Resolve each program to a real `.exe`; reject any resolved target ending `.bat`, `.cmd`, `.ps1`, `.com`, `.vbs`.
2. Set `windowsVerbatimArguments: false` (default) and `windowsHide: true`; never enable `shell`.
3. Run a patched Bun; treat npm-shipped `.cmd` wrappers (e.g. `eslint.cmd`) as **not allowlisted** — invoke the underlying JS entrypoint via `bun run`/`node` or resolve to the real binary.

## C.3 Filesystem confinement to repo root

A single `resolveInsideRepo()` chokepoint used by **every** path any tool touches (read_file, list_dir, grep glob, and the spawn `cwd`). **Dereference symlinks before the boundary check** — validating first then following links is the classic bypass.

```ts
import { realpathSync } from 'node:fs';
import { resolve, sep, isAbsolute } from 'node:path';

const REPO_ROOT = realpathSync(resolve(process.env.REPO_ROOT!)); // canonical root, once

export function resolveInsideRepo(userPath: string): string {
  // 1. Reject absolute, Windows drive, and UNC paths outright.
  if (isAbsolute(userPath) || /^[a-zA-Z]:[\\/]/.test(userPath) || userPath.startsWith('\\\\'))
    throw new SafetyError('path_outside_repo', 'absolute paths are not allowed');
  // 2. Resolve relative to root (collapses '..' and '.').
  const candidate = resolve(REPO_ROOT, userPath);
  // 3. Canonicalize, following symlinks (for a not-yet-existing path,
  //    realpath the nearest existing ancestor instead).
  const real = realpathSyncNearestExisting(candidate);
  // 4. Boundary check on the CANONICAL path. Trailing sep stops
  //    '/repo' matching a sibling '/repo-evil'.
  if (real !== REPO_ROOT && !real.startsWith(REPO_ROOT + sep))
    throw new SafetyError('path_outside_repo', 'path escapes repository root');
  return real;
}
```

Rules: no absolute/drive/UNC paths; `..` collapsed by `resolve` then boundary-checked; symlink escapes fail because `realpath` runs _before_ `startsWith`; `list_dir`/`grep` do not follow symlinks outward; the spawned `cwd` is pinned to canonical `REPO_ROOT`.

## C.4 Where the boundary lives: server-side only

```
model ──proposes tool_call──▶ [ SERVER VALIDATOR ] ──▶ executor (Bun.spawn / Bun.file)
                                     │
              (1) tool exists?       │  reject → structured error back to model
              (2) path confined?     │
              (3) allowlist match?   │  ← single source of truth
              (4) denylist clear?    │
              (5) metachar scan?      │
```

- The validator is the **only** code path to execution. No tool bypasses it.
- The model's proposed `argv`/`path` is **data**, never spliced into a string. Executor builds `Bun.spawn({ cmd: [program, ...validatedArgs], cwd: REPO_ROOT, env: MINIMAL_ENV, ... })`.
- **Minimal env:** only `PATH` (trimmed to the toolchain dir), `HOME`/`USERPROFILE`, `SYSTEMROOT`. Forward **no** secrets (`GITHUB_TOKEN`, `AWS_*`, npm tokens) — blunts exfiltration and stops env-driven config redirecting writes.
- System-prompt "you are read-only" is **advisory only**. If prompt and validator disagree, the validator wins.

## C.5 Failure modes, timeouts, caps, graceful rejection

```ts
const proc = Bun.spawn({
  cmd: [program, ...args], // validated argv array — NO shell
  cwd: REPO_ROOT,
  env: MINIMAL_ENV,
  stdin: 'ignore', // model cannot feed stdin
  stdout: 'pipe',
  stderr: 'pipe',
  timeout: 30_000, // wall-clock cap → auto-kill
  killSignal: 'SIGKILL', // hard kill on timeout/abort
  signal: abortController.signal, // user can cancel from UI
  windowsHide: true,
  // windowsVerbatimArguments: false (default), shell NOT set
});
```

- **Timeout:** `timeout` auto-terminates with `killSignal` (`spawnSync` exposes `exitedDueToTimeout`).
- **Output cap:** stream stdout/stderr, stop at a byte cap (~1 MiB), kill, return truncated output flagged `"truncated": true` (`spawnSync` has native `maxBuffer`).
- **Concurrency:** one in-flight `run_command` per session; small queue; reject floods.
- **Graceful rejection:** every denial returns a **structured tool result**, not a thrown error — categories `command_not_allowed`, `flag_not_allowed`, `path_outside_repo`, `is_a_symlink_escape`, `output_truncated`, `timed_out`, `binary_file`, `too_large` — with a `reason` and an `allowed` hint so the model picks a legal alternative.
- **Fail closed:** any unexpected validator error → deny. Never "allow on error."

## C.6 Optional hardening (defense-in-depth)

For a private repo the validator + no-shell + confinement is the core guarantee. To lower the blast-radius floor (Oligo, Trail of Bits, SandboxEval 2025): run the executor under an OS sandbox so even an exploitable allowlisted tool can't escape — **Windows:** restricted token / AppContainer / Windows Sandbox with the repo mounted and the rest of FS read-only or absent, outbound network denied; **cross-platform:** container/microVM with the repo bind-mounted **read-only at the OS level**, no network namespace, CPU/mem limits.

## C.7 Validator — TypeScript sketch (Bun)

```ts
import { realpathSync } from 'node:fs';
import { resolve, sep, isAbsolute } from 'node:path';

class SafetyError extends Error {
  constructor(
    public code: string,
    msg: string,
    public meta: object = {},
  ) {
    super(msg);
  }
}
const REPO_ROOT = realpathSync(resolve(Bun.env.REPO_ROOT!));

const SHELL_METACHARS = /[;&|<>`$(){}]|&&|\|\||\n|\r|>>|2>|&>/;
const REDIRECT_TOKENS = new Set(['>', '>>', '1>', '2>', '&>', '<', '|']);
const BATCH_EXT = /\.(bat|cmd|ps1|com|vbs)$/i;

type Rule = { subs?: Record<string, FlagRule>; flags?: Set<string> | 'ANY' };
type FlagRule = { flags?: Set<string> | 'ANY'; argAllowlist?: Set<string> };

const ALLOW: Record<string, Rule> = {
  git: {
    subs: {
      log: { flags: 'ANY' },
      status: { flags: 'ANY' },
      diff: { flags: 'ANY' },
      show: { flags: 'ANY' },
      blame: {},
      'ls-files': {},
      'ls-tree': {},
      'rev-parse': {},
      'cat-file': {},
      shortlog: {},
      branch: { flags: new Set(['--list', '-a', '-v']) },
    },
  },
  bun: {
    subs: {
      test: { flags: 'ANY' },
      run: { argAllowlist: new Set(['check', 'typecheck', 'tsc', 'lint', 'bench']) },
      pm: { argAllowlist: new Set(['ls']) },
    },
  },
  rg: { flags: 'ANY' },
  tsc: { flags: 'ANY' },
  prettier: { flags: new Set(['--check']) }, // never --write
  eslint: { flags: new Set(['--format', '--max-warnings', '.']) }, // never --fix
  oxlint: { flags: 'ANY' },
};

const DENY_TOKENS = new Set([
  'push',
  'commit',
  'add',
  'pull',
  'fetch',
  'merge',
  'rebase',
  'reset',
  'checkout',
  'switch',
  'restore',
  'clean',
  'rm',
  'mv',
  'stash',
  'apply',
  'gc',
  'prune',

  'unlink',
  'patch',
  'create',
  '--fix',

  '-i',
  '--in-place',
  '-o',
  '--output',
  '--output-file',
  '--log-file',
]);
const DENY_PROGRAMS = new Set([
  'bash',
  'sh',
  'zsh',
  'cmd',
  'cmd.exe',
  'powershell',
  'pwsh',
  'wsl',
  'env',
  'rm',
  'del',
  'erase',
  'mv',
  'move',
  'cp',

  'ren',
  'rename',
  'mkdir',
  'touch',
  'ln',
  'dd',
  'tee',

  'chown',
  'attrib',
  'icacls',
  'curl',
  'wget',
  'nc',
  'ssh',
  'scp',
  'rsync',
  'npm',
  'npx',

  'yarn',
  'pip',
  'cargo',
  'go',
  'make',
]);

export function validateCommand(argv: string[]): string[] {
  if (!Array.isArray(argv) || argv.length === 0)
    throw new SafetyError('command_not_allowed', 'empty command');
  const [program, ...args] = argv;

  if (program.includes('/') || program.includes('\\') || BATCH_EXT.test(program))
    throw new SafetyError('command_not_allowed', 'program must be a bare executable name');
  if (DENY_PROGRAMS.has(program.toLowerCase()))
    throw new SafetyError('command_not_allowed', `${program} is a write/network/shell tool`);

  const rule = ALLOW[program];
  if (!rule) throw new SafetyError('command_not_allowed', `${program} is not on the allowlist`);

  for (const a of argv) {
    if (DENY_TOKENS.has(a.toLowerCase()))
      throw new SafetyError('command_not_allowed', `disallowed token: ${a}`);
    if (REDIRECT_TOKENS.has(a) || SHELL_METACHARS.test(a))
      throw new SafetyError('command_not_allowed', `shell metacharacter/redirection: ${a}`);
    if (/^[A-Za-z_][A-Za-z0-9_]*=/.test(a) && a === program)
      throw new SafetyError('command_not_allowed', 'env-var assignment not allowed');
  }

  if (rule.subs) {
    const sub = args[0];
    const fr = sub ? rule.subs[sub] : undefined;
    if (!fr) throw new SafetyError('command_not_allowed', `${program} ${sub ?? ''} not allowed`);
    const rest = args.slice(1);
    if (fr.argAllowlist)
      for (const a of rest)
        if (!a.startsWith('-') && !fr.argAllowlist.has(a))
          throw new SafetyError('command_not_allowed', `${program} ${sub} ${a} not allowed`);
    if (fr.flags && fr.flags !== 'ANY')
      for (const a of rest)
        if (a.startsWith('-') && !fr.flags.has(a))
          throw new SafetyError('flag_not_allowed', `flag ${a} not allowed for ${program} ${sub}`);
  } else if (rule.flags && rule.flags !== 'ANY') {
    for (const a of args)
      if (a.startsWith('-') && !rule.flags.has(a))
        throw new SafetyError('flag_not_allowed', `flag ${a} not allowed for ${program}`);
  }
  return argv; // validated; safe to spawn as an ARRAY (no shell)
}

export async function runValidated(argv: string[], ac: AbortController) {
  const cmd = validateCommand(argv);
  const proc = Bun.spawn({
    cmd,
    cwd: REPO_ROOT,
    env: MINIMAL_ENV,
    stdin: 'ignore',
    stdout: 'pipe',
    stderr: 'pipe',
    timeout: 30_000,
    killSignal: 'SIGKILL',
    signal: ac.signal,
    windowsHide: true,
  });
  const out = await readCapped(proc.stdout, 1 << 20); // 1 MiB cap → kill+truncate
  const code = await proc.exited;
  return { ok: true, exitCode: code, truncated: out.truncated, stdout: out.text };
}
```

## C.8 Load-bearing decisions

1. Four read/run-only tools; no write capability exists. `run_command` takes an **argv array, never a string**.
2. **Default-deny allowlist** keyed by `(program, subcommand, flags)` + an independent **denylist/metachar/redirect scan** as a second wall; `bun run` limited to a script-name allowlist; `prettier --check`/`eslint`/`oxlint` with no `--fix`/`--write`.
3. **`realpath`-then-boundary-check** confinement at one chokepoint used by every path and the spawn `cwd`; absolute/drive/UNC and symlink escapes rejected.
4. **Windows CVE-2024-27980 hardening:** never spawn `.bat`/`.cmd`/`.ps1`; `shell` never enabled; `windowsHide`, default verbatim-args.
5. **Server is the sole authority** — model proposes, validator+executor decide; prompt instructions are not a control.
6. **Bounded execution:** `timeout`+`killSignal`, output byte caps, minimal env (no secrets), `stdin: "ignore"`, AbortSignal cancel, structured fail-closed errors. Optional OS sandbox / read-only mount for true containment.

# D. Phased, Risk-Ordered Implementation Plan

This plan covers **every** requested feature. For each: concrete approach, files to touch (from the repo map), determinism/perf risks, gate impact (`bun run check` must stay green: prettier, tsc strict, oxlint, bun test, build), and the recommended WAVE. Features needing the AI token are flagged with graceful-degradation behavior.

## D.0 The Gate (`bun run check`) — what every wave must keep green

Five sub-gates: **prettier** (format), **tsc --strict** (no `any` leaks, exact-optional, null-safety), **oxlint** (no unused, no `Math.random` in sim — add a custom guard if not present), **bun test** (determinism/parity tests), **build** (the Vite/Bun bundle compiles). Per-wave gate rules:

- New systems must be **rng-parity-neutral** or constructed after `atmosphere` (the last rng consumer) so existing replay seeds still reproduce.
- Every new typed-array field must be initialized for the full pool (tsc strict + parity tests catch holes).
- Any new stochastic line must consume `this.rng()` (the seeded stream), never `Math.random` — add/extend a parity test that runs N frames twice from one seed and asserts identical telemetry hashes.
- Prettier/oxlint are mechanical; budget a format pass before committing each wave.

## D.1 Determinism & perf risk legend

- **D-LOW:** no new RNG draws, pure per-frame physics or appearance.
- **D-MED:** new RNG draws but localized; needs a parity test and seeded-stream routing.
- **D-HIGH:** touches reproduction/selection/iteration order — the make-or-break zone (§A.10).
- **P-LOW / P-MED / P-HIGH:** per-frame cost at 10k entities.

### F-NHI · User-launched NHI beings on click ("Matrix powers")

- **Approach:** In `src/ui/input.ts` (~407–443) distinguish click from look-drag (movement threshold); on a clean click, ray-cast camera→ground plane (`THREE.Raycaster` + `Plane(y=0)`) and call a new `spawn` action. ### F-HOLES · Holes/singularities affect the world incl. big objects + environment artifacts

- **Approach:** Publish the active singularity's position/kind into `SimState` (e.g. `s.activeSingularityPos`). - **Files:** `src/sim/singularities.ts` (~180–230), `src/sim/shoggoths.ts`, `src/sim/titans.ts`, `src/sim/puppet-masters.ts`, `src/sim/reaction-diffusion.ts`, `src/types.ts` (SimState field), `src/world.ts` (publish + update order — singularities already update before entities ~419).
- **Risk:** D-MED — force application is deterministic float math, but **fix the order**: publish singularity state before the big-being updates each frame, and accumulate forces in fixed index order. The entropy singularity already raises chaos (`singularities.ts` ~223); keep that. P-MED (a few extra distance checks for ~16 big beings — negligible; RD-coupling is the GPU cost, already paid).
- **Gate:** parity test summoning each singularity kind from a fixed seed and hashing big-being trajectories.

### F-CHAOS-ENTROPY · Chaos levels + entropy (bipolar axis)

- **Approach:** Two designs — (a) keep `chaos ∈ [0.1,10]` and add a separate `entropy` scalar with its own decay and effects (freeze/uniformity/greyscale), or (b) unify into `chaosEntropy ∈ [-10,+10]` (positive=chaos/agitation, negative=entropy/heat-death, 0=equilibrium). Recommend (a) first (additive, lower churn): add `s.entropy` to `SimState`, decay toward 0, drive it up via the entropy singularity and idle uniformity, and apply effects (reduced jitter, desaturated atmosphere) bidirectionally. Add a second control/slider in `index.html`.
- **Files:** `src/types.ts` (SimState), `src/sim/constants.ts` (bounds), `src/world.ts` (decay/floor/coupling ~382–392, action ~1171), `src/sim/singularities.ts` (entropy coupling ~219), `src/sim/atmosphere.ts` (visual coupling), `index.html` + `src/ui/input.ts` (control).
- **Risk:** D-MED — decay/floor are deterministic, but any entropy-driven _stochastic_ effect (e.g. random freeze) must use the seeded stream. Chaos already affects behavior jitter (`chaosMul()` ~542) and sort push (~666); mirror carefully. P-LOW.
- **Gate:** parity test over a fixed chaos/entropy script; ensure new SimState field is serialized in any save/replay.

**Wave 2 rationale:** high "wow" (player agency + cosmology) at D-MED; each is independently shippable and guarded by an audit-replay parity test established in Wave 1.

- **Risk:** D-LOW (deterministic GPU passes with strict double-buffering), P-MED (a few extra texture passes; 128² is cheap). Blackboard writes in fixed priority order.
- **Gate:** parity test hashing influence-map readback after N frames; keep new context fields typed.

### F-AI-MICRO · Utility/needs arbiter + boids steering + tiny genome-MLP (§A.4, A.5, A.9)

- **Approach:** Add a per-organism **needs vector** (typed arrays) and a utility scorer over the 26 behaviors using response-curve LUTs + smart-object advertisements (sample influence maps). Wrap movement in boids/steering reusing the existing spatial hash. Formalize the connectome as a fixed genome-encoded MLP whose forward pass feeds behavior logits. Add a **fuzzy mood** smoothing layer and a per-species **decision tree**.
- **Files:** `src/sim/entities.ts` (needs arrays, scorer, integration), `src/sim/behaviors.ts` (behaviors become leaf actions; steering forces), `src/sim/connectome.ts` (MLP forward), `src/math/spatial-hash.ts` (reuse), `src/math/scalar.ts` (response curves), `src/world.ts` (wiring/cadence).
- **Risk:** D-HIGH-adjacent — softmax/weighted action sampling and any tie-break **must** draw from the seeded stream; **accumulate boid neighbor forces in fixed index order** (the #1 flocking determinism pitfall, §A.5). P-MED at 10k (utility is K MACs+max; MLP ~70 MACs; both cheap — but profile against the connectome-rebuild cadence ladder ~line 441).
- **Gate:** the critical parity test — two 1000-frame runs from one seed, assert identical telemetry + position hashes; bun-test must stay green. Budget the most gate effort here.

### F-AI-MACRO · Tribe HFSMs + GOAP titans + Markov regimes (§A.1, A.3, A.7)

- **Approach:** Make Louvain tribes HFSMs (Peaceful→Alarmed→Migrating→War) biasing member transitions. Give the 10 titans a small GOAP planner whose cost function is the existing iterated-PD/replicator output (`src/math/games.ts`); cache plans, replan only on macro-state change, budget replans/frame. Markov transition matrices drive weather/economy regimes.
- **Files:** `src/sim/titans.ts` (GOAP + economy ~594–780), `src/math/games.ts` (cost functions), tribe/Louvain code (graph-mind pass ~world.ts 460), `src/sim/weather` (Markov regimes), `src/sim/blackboard.ts` (intents).
- **Risk:** D-HIGH — GOAP A\* tie-breaks by stable action index (never `Map` order); Markov draws from the seeded stream; replicator/fitness summation in fixed order; **stable-sort titans/tribes before any selection**. P-MED (plans tiny + cached; 10 titans is nothing).
- **Gate:** parity test over a long macro run; assert titan strategy census and tribe states reproduce.

**Wave 3 rationale:** highest design value but D-MED→D-HIGH; built strictly bottom-up (perception → micro → macro) so each sub-wave keeps the gate green and the parity tests accrete.

## WAVE 4 — Evolution, lineage & sentience feedback (highest determinism risk)

### F-LINEAGE · Offspring / lineage / social-economy + rtNEAT (§A.10)

- **Approach:** Extend `EntityData` (`src/types.ts` ~51–94) with `parentId`, `generation`, and the genome (MLP weights). Reproduction in `doSplit()` (`src/world.ts` ~756–777) / `split` behavior (`src/sim/behaviors.ts`) becomes **crossover + seeded mutation**; pass parent id. Adopt **rtNEAT continuous replacement** (replace one low-fitness agent per K ticks) so lineage/speciation emerge without generational stalls. Wire fitness = lifespan + energy + offspring + tribe contribution. Tie species≈tribe to Louvain. Extend the titan economy to consume/produce against lineage success.
- **Files:** `src/types.ts`, `src/sim/entities.ts` (spawn with parentId, genome arrays), `src/sim/behaviors.ts`, `src/world.ts` (doSplit, rtNEAT replacement pass, fitness), `src/sim/titans.ts` (economy coupling), `src/sim/connectome.ts` (genome→MLP), a new lineage/kinship pass (reuse graph-mind).
- **Risk:** **D-HIGH — the make-or-break zone.** All selection, crossover points, mutations from the single `mulberry32` stream; **stable-sort population by a stable key before selection** (never `Set`/`Map` order); a fixed innovation-number counter; reproducible fitness summation order. P-MED (inference cheap; evolution amortized to ~1 birth/K frames — cap topology growth to bound the MLP forward cost and the connectome-rebuild ladder).
- **Gate:** the toughest parity test — reproduce an entire lineage tree over a long seeded run and assert identical genealogy hashes. Budget the most QA here; this is where the global RNG silently desyncs.

### F-SENTIENCE-FEEDBACK · Sentience variation coupled to dynamics

- **Approach:** Promote the read-only metric from Wave 1 to a driver: high mean activation / link-density nudges chaos, mutation rate, or behavior thresholds (per-species variation in "awareness"). Gate behind a flag so it can be toggled for parity comparisons.
- **Files:** `src/world.ts`, `src/sim/entities.ts`/`behaviors.ts`, `src/sim/connectome.ts`.
- **Risk:** D-HIGH (feeds back into stochastic systems) — route all coupling through seeded draws; parity test with the flag on. P-LOW.
- **Gate:** parity test on; ensure deterministic sampling cadence.

**Wave 4 rationale:** maximal payoff (true evolution + lineage) but the riskiest for determinism — deliberately last among sim features so all parity infrastructure from Waves 1–3 is in place to catch desyncs.

## WAVE 5 — More being types & the AI chat sidecar

### F-BEINGS · More being types (beyond shoggoths/puppets/titans)

- **Approach:** Use the TitanSystem as the template. Create e.g. `src/sim/leviathan-swarm.ts` (a roaming collective) or `src/sim/orbital-fortress.ts` with a class taking `(ctx, entities, …)`, a state-array pool, geometry built once, a roaming/collective `update(dt,t)`, harvest/scavenge on a `frame % N` cadence, spatial queries against the grid, and a `count` getter for telemetry.

- **Gate:** ensure new telemetry fields are typed; parity test if rng-touching. Integrate the singularity-affects-bigs (F-HOLES) hook for new beings.

### F-AI-CHAT · Free-AI read/run chat sidecar (Part B backend + Part C safety) — NEEDS SECRET

- **Approach:** Stand up a Bun server (separate from the sim build) implementing the Part C validator + executor and the Part B GitHub Models proxy. The sim's `/lab` link (`index.html:941`) gains a chat panel that talks to the sidecar. The model proposes `read_file`/`list_dir`/`grep`/`run_command`; the server validates, runs read-only against the repo, returns results; chat answers repo questions and can run `bun run check` read-only.
- **Files:** new `server/sidecar.ts` (proxy + validator + executor), new `server/validator.ts` (Part C sketch), a chat UI panel (new `src/ui/chat.ts` + markup near the `/lab` link), `package.json` (a `sidecar` script — NOT in `bun run check`'s allowlisted scripts to avoid the chat self-invoking writes).
- **Secret:** **requires `GITHUB_TOKEN`** (fine-grained PAT, `models: read`), server-side only, never bundled to the browser.
- **Graceful degradation when the token is absent:** the sidecar starts but the chat panel shows a clear "AI chat disabled — set `GITHUB_TOKEN` to enable" state; the `read_file`/`list_dir`/`grep`/`run_command` tools still work for a local/manual mode if desired, but no model calls are made. The **sim itself never depends on the token** — it is a pure sidecar; the main `bun run check` and the WebGL app build and run with zero AI configuration. If the token is present but rate-limited (50–150 req/day), surface the structured error and back off.
- **Risk:** D-N/A (outside the deterministic sim — the sidecar must never write to sim state or the repo). P-N/A (separate process). The real risk is **security**, fully addressed by Part C (untrusted model, server-side allowlist, no shell, FS confinement, CVE-2024-27980, minimal env, timeouts/caps).
- **Gate:** the sidecar is a separate target — keep it out of the WebGL bundle so `build` stays clean; its own `tsc`/`oxlint`/tests run under the same `bun run check` umbrella. Critically, the sidecar's `run_command` allowlist permits `bun run check` (read-only) but `bun run` is restricted to the script-name allowlist so the chat can't trigger a writing script.

**Wave 5 rationale:** more being types are low-risk additive content; the chat sidecar is deferred last because it is the only feature needing a user secret and it is architecturally isolated from the deterministic core — shipping it last means the sim is fully complete and green before any AI dependency is introduced, and its absence degrades to nothing.
